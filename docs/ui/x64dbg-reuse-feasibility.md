# x64dbg UI 复用可行性与边界评估

用于任务 3.1/7.3：评估“先复用，后重写”策略。

## 结论

- 推荐策略：**仅复用设计思路与交互映射，不直接绑定 x64dbg 原生 UI 库作为长期依赖**。
- 落地方式：
  - Phase A：通过配色、分区、控件语义快速做同构验证；
  - Phase B：在本项目 UI 层完成自研实现并移除过渡依赖。

## 技术维度

- x64dbg 生态与本项目 Rust + egui 技术栈不一致，直接调用原生 UI 库的跨平台成本高。
- 通过 `src/ui/layout` + `src/ui/style` 抽象层，可将同构约束与具体实现解耦。
- 当前实现已将面板 ID、布局比例、主题 token 固化，便于自动化回归。

## 兼容性维度

- macOS 双架构目标（x64/arm64）需要避免平台绑定型 UI 依赖。
- 采用纯 Rust/egui 方案更利于 CI 构建稳定性和后续维护。

## 许可与维护维度

- 直接引入第三方 UI 资产需额外审查许可条款与再分发风险。
- 维护上，长期绑定外部 UI 组件会增加升级耦合和安全修复成本。

## 对应关系（x64dbg → 本项目）

| x64dbg 概念 | 本项目模块 | 说明 |
|---|---|---|
| 主停靠布局 | `src/ui/layout/mod.rs` | 固化分区与比例 |
| 状态配色 | `src/ui/style/mod.rs` | 统一主题 token |
| 控制栏附加入口 | `src/ui/control_panel.rs` | PID/进程名附加 |
| 状态/错误栏 | `src/ui/status_bar.rs` | 附加生命周期与错误分类 |

## 差异说明

- 非目标：像素级 1:1 还原；
- 目标：操作路径、信息架构、状态反馈保持一致；
- 差异需记录：若控件行为因框架限制不同，必须在 PR 描述中说明并附截图。
